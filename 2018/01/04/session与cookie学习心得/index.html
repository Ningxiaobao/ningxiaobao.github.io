session 和 cookie

cookie

cookie 是存储在客户端上的一小段数据，浏览器通过HTTP协议和服务器进行cookie交换。

cookie 独立与语言存在，但很多语言都有实现 cookie [ 其实：间接操作（发生 HTTP 指令）]，因此 cookie 是由浏览器实现和管理的

例：PHP 可以设置 cookie，但实际上 PHP 并没有实际的设置过 cookie，可以说PHP没有能力设置 cookie ，PHP 只是发出命令让浏览器来做这件事情

cookie 一般存放不敏感信息，而且 cookie 的数量不是越多越好，它会增加带宽。

cookie 一般会保存到两种地方：1.文件中 2.浏览器内存中

正常的 cookie 只能在一个应用中共享（即：一个 cookie 只能由创建他的应用获得），但为了实现 cookie 的跨域使用，最好的方式就是用 P3P 协议

一个域名的每个 cookie 限制以 4 千字节的键值对的形式存在，若是服务端也使用的话，cookie 的存储大小就可能不够用，于是出现一个新的解决方案：localstorage。[在这里我就不详述了]

session

session 即会话，一种持续性的、双向性的连接

session 的实现方式：URL 重写、cookie、通过在 cookie 中存储的 sessionID 实现 session 传递

session 指用户进入网站到浏览器关闭这段时间内的会话。[特定的时间概念]

以 PHP 为例：session 默认以文件方式实现，存储在服务器的 session 文件，每个 session 一个文件

session 是通过 sessionID 判断客户端用户的，即：session 文件的文件名

sessionID 必须包含在 HTTPRequest 里面，保证唯一性和随机性。

没有设置 session 的生命周期，sessionID 存储在内存中，关闭浏览器后 ID 自动注销，若设置生命周期，session 过期后，才会回收。session 并不是都随浏览器关闭而关闭。

易错点

  1. cookie 运行在客户端 session 运行在服务器端对吗？

    答：不完全正确，因为 sessionID 作为 cookie 存储在客户端

  2. 浏览器禁用 cookie，cookie 不能用了,session 会受到影响吗？

    答：session 会受到影响，但可以通过 URL 传递 session

  3. session 比 cookie 更安全吗？

    答：cookie 和 session 是绑定的，获得 cookie 就相当获得了 session，客户端吧劫持的 cookie 原封不动地传给服务器，服务器收到后，原封不动的验证session，若 session 存在，就实现了 cookie 和 session 的绑定过程。因此，不存在 session 比cookie 更安全的说法。

  4. 如果把别人机器上的 cookie 文件复制到我的电脑上，是不是可以登陆别人账号。

    答：可以，这样的操作属于 cookie 劫持，要避免这种情况，需要在 cookie 中针对 IP、UA 等加上特殊的校验信息，然后再和服务器对比。
